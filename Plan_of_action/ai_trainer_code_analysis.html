Critical
Transcription overwrites the next question's answer
InterviewSession.tsx  sendAudioForTranscription / handleNextQuestion
›

sendAudioForTranscription() is fire-and-forget. After the user clicks "Next Question", saveCurrentAnswer() runs immediately. But the cloud STT response arrives 2–5 seconds later and calls setCurrentAnswer(prev => …) — which now patches the new question's answer box, not the old one. The wrong answer gets saved and the user's typed corrections are overwritten.

// ❌ current — fire and forget, no question context captured recorder.onstop = () => { stream.getTracks().forEach(t => t.stop()); sendAudioForTranscription(); // runs async, has no idea which Q we're on };
Fix: Snapshot currentIdx before the async call and only apply the transcript if the index hasn't changed. Or better: use a questionId ref to guard the state update.

const qIdAtRecordStop = currentQuestion.id;
// …in sendAudioForTranscription:
if (currentQuestion?.id === qIdAtRecordStop) setCurrentAnswer(…);
Critical
Answer saved before transcription completes
InterviewSession.tsx  handleNextQuestion → saveCurrentAnswer
›

handleNextQuestion() calls saveCurrentAnswer() synchronously and then advances. Cloud STT takes 2–5 seconds. The answer collected into collectedAnswers[idx] is whatever partial text existed at click time — usually empty or cut short.

const handleNextQuestion = () => { stopSpeaking(); if (isRecording) stopRecording(); saveCurrentAnswer(); // ← transcript not ready yet if (currentIdx + 1 >= questions.length) setPhase('review'); else { setCurrentIdx(prev => prev + 1); … } };
Fix: Disable the "Next Question" button while isTranscribing === true, or wait for transcription to resolve before advancing. The button already has disabled={isRecording} — extend it to disabled={isRecording || isTranscribing} and add a "Transcribing…" label.
Critical
localStorage session key is not user-scoped
InterviewSession.tsx  STORAGE_KEY constant
›

The backup key is a fixed string 'interview_session_backup'. If two users share the same browser (common in labs or family computers), User B logs in and the mount useEffect finds User A's saved session, restores it, and skips startInterview() entirely. User B is now in User A's interview.

const STORAGE_KEY = 'interview_session_backup'; // ❌ shared across all users
Fix: Append the logged-in user's ID to the key. Read the user ID from AuthService.getUser()?.id:

const userId = AuthService.getUser()?.id ?? 'anon';
const STORAGE_KEY = `interview_session_backup_${userId}`;
Critical
Only ValueError caught in submit_all — unhandled 500s
apps/interview/views.py  submit_all view
›

The evaluation call is wrapped in except ValueError only. Gemini JSON parse errors (json.JSONDecodeError), network timeouts, AttributeError from a bad response shape, or quota errors from key rotation failure all propagate as unhandled Django 500 errors — the user sees a blank error instead of a friendly retry message.

try: evaluation = evaluate_interview(answers) except ValueError as e: # ❌ too narrow return Response({'error': 'Interview evaluation failed.'}, status=503)
Fix: Catch Exception and log the specific type for debugging:

except Exception as e:
    logger.error("submit_all failed [%s]: %s", type(e).__name__, e)
    return Response({'error': 'Evaluation failed. Please try again.'}, status=503)
Critical
Audio file read fully into memory before size check
apps/interview/views.py  transcribe_audio
›

A malicious user (or broken client) can send a 500 MB audio file. The entire payload is read into a Python bytes object before the 10 MB check runs — it's too late, the RAM spike already happened.

audio_bytes = audio_file.read() # ❌ loads ALL bytes first if len(audio_bytes) > MAX_AUDIO_BYTES: # then rejects return Response({"error": "Too large."}, status=400)
Fix: Check the file size before reading using the Content-Length header or Django's audio_file.size attribute (set at upload time):

if audio_file.size > MAX_AUDIO_BYTES:
    return Response({"error": "Audio file too large."}, status=400)
audio_bytes = audio_file.read()

›

question_number comes straight from the request body with no bounds check. A client can send "question_number": 999 and the view will immediately return next_question: null, is_last: true — completing the interview with zero real answers answered.

actual_answered = InterviewQuestion.objects.filter(session=session).count()

The Zustand store initialises with user: null, token: null on every cold load. Medium
generate_questions hard-fails on partial Gemini response
services/openai_service.py  generate_questions
›

If Gemini returns 7 valid questions instead of 8 (a common occurrence with token truncation), the function raises ValueError("Gemini returned 7 valid questions, expected 8"). The entire start_interview view returns 503 and the user cannot start an interview.

if len(validated) < num_questions: raise ValueError(f"Gemini returned {len(validated)} …") # ❌ no recovery
Fix: Accept partial results if they meet a minimum threshold, and backfill with fallback questions if needed:

if len(validated) < max(3, num_questions - 2):
    raise ValueError(…) # only fail if very few returned
return validated[:num_questions] # accept 7/8, pad if needed

Medium
LiveInterviewSession VAD finalizeAnswer double-call race
pages/LiveInterviewSession.tsx  startRecording / finalizeAnswer
›

The requestAnimationFrame VAD loop checks phaseRef.current !== 'recording' as a guard, but setPhase('processing') is asynchronous — it doesn't update phaseRef.current until the next React commit. A second RAF tick can fire between the first finalizeAnswer() call and the ref update, calling it twice and starting two STT→backend calls for the same question.

Fix: Add a local ref flag that is set synchronously before any async work:

const finalizeCalledRef = useRef(false);

const finalizeAnswer = useCallback(async () => {
  if (phaseRef.current !== 'recording') return;
  if (finalizeCalledRef.current) return; // ← add this
  finalizeCalledRef.current = true; // ← and this
  setPhase('processing');
  …
}, […]);

Reset finalizeCalledRef.current = false at the start of each askQuestion() call.

Exception gap
cloud_stt_service has no retry on transient errors
apps/interview/services/cloud_stt_service.py
›

The Google Cloud STT call is a single attempt with no retry. A brief 503 or network hiccup from Google surfaces directly as a failed transcription. The user's recorded audio is lost and they must re-record.

Fix: Add exponential backoff retry for transient errors (503, 429, network timeout):

import time
for attempt in range(3):
    try:
        response = client.recognize(request=request)
        break
    except Exception as e:
        if attempt == 2: raise
        time.sleep(0.5 * (2 ** attempt))
Exception gap
cloud_tts_service has no fallback voice
apps/interview/services/cloud_tts_service.py
›

If en-IN-Chirp3-HD-Aoede is unavailable (quota exhausted, region not available), the service raises immediately with no fallback. The entire /api/interview/tts/ endpoint returns 503 and the frontend falls back to browser TTS — but only because of the try/catch in InterviewSession.tsx, not because the backend handles it gracefully.

Fix: Try a list of fallback voices in order:

VOICES = ["en-IN-Chirp3-HD-Aoede", "en-IN-Standard-A", "en-US-Standard-B"]
for voice_name in VOICES:
    try:
        return synthesize_speech(text, voice_name)
    except Exception:
        continue
raise RuntimeError("All TTS voices failed")

InterviewFeedback useEffect has no cleanup — setState on unmount
pages/InterviewFeedback.tsx  fetchFeedback useEffect
›

If the user navigates away from the feedback page while getFeedback() is in-flight, the response eventually arrives and calls setSessionData() and setFeedbackData() on an already-unmounted component — a React warning and potential null-pointer error in strict mode.

useEffect(() => {
  let active = true;

  return () => { active = false; };
}, [routeSessionId]);

ai_trainer/settings.py
›

The default SECRET_KEY is a known hardcoded string published in this repository. If the .env file is missing (e.g., on a new deployment), Django silently runs with this key. Anyone with the source code can forge session cookies and CSRF tokens.

SECRET_KEY = config('SECRET_KEY', default='django-insecure-r@jw1!^&#k%)04…') # ❌ published default

SECRET_KEY = config('SECRET_KEY') # raises ImproperlyConfigured if missing

The evaluation call is wrapped with a generic except Exception that logs but hides the actual error type. A KeyError in score parsing, a DB constraint violation, and a Gemini quota error all produce the same generic "Evaluation failed. Please try again." — impossible to debug from logs alone.

except Exception as e:

pages/InterviewSession.tsx  startInterview / handleSubmitAll
›

Both startInterview() and handleSubmitAll() are async functions that call apiPost() (which internally uses fetch). If the user navigates away mid-request, the fetch completes and calls setPhase(), setQuestions(), etc. on a dead component.

Improvement
VAD calibration: use median-based threshold instead of mean+2σ
pages/LiveInterviewSession.tsx  startRecording VAD
›

The current calibration uses mean + stdDev * 2. A single loud noise during the 1.2s calibration window (a cough, a door slam) inflates the mean and pushes the threshold so high that normal speech never clears it — the interview listens forever and never auto-advances.

Fix: Use the median of calibration samples instead of the mean — it's resistant to outliers:

const sorted = [...baselineSamples].sort((a,b)=>a-b);
const median = sorted[Math.floor(sorted.length/2)];
dynamicThreshold = Math.max(8, median * 2.5);
