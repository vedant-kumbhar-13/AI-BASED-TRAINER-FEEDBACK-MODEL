[Cell 0]: 🔍  Symptom
[Cell 1]: During voice recording, if the user clicks "Next Question" within 200ms of the silence timer firing, handleNextQuestion() runs twice in rapid succession. The second call saves the answer as "[No answer provided]" and skips to the question after the intended one. The user loses Q+1 answer and jumps to Q+2 without realising.

[Cell 0]: 🧠  Root Cause
[Cell 1]: The silence timer (silenceTimerRef) fires after 3 seconds. Its callback calls recognition.stop(), then launches an anonymous setTimeout(() => handleNextQuestion(), 200). This inner 200ms timeout is not stored in any ref. stopListening() correctly clears silenceTimerRef, but cannot cancel the inner timeout because it has no reference. If the user clicks Next in that 200ms window, handleNextQuestion() fires from both the button click and the orphaned timeout.

[Cell 0]: 🛠  Approach
[Cell 1]: Introduce a second ref (advanceTimerRef) to hold the 200ms timeout. Clear it at the start of stopListening() and at the start of handleNextQuestion() — wherever the advance is initiated first wins, and the second attempt is a no-op.

[Cell 0]: ✏️  Add the ref (after silenceTimerRef)
[Cell 1]: // After line ~103 where silenceTimerRef is declared:
✅ const advanceTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);

[Cell 0]: ✏️  Fix the silence timer callback (~line 217)
[Cell 1]: // BEFORE:
❌ silenceTimerRef.current = setTimeout(() => {
❌   intentionalStopRef.current = true;
❌   recognitionRef.current?.stop();
❌   setTimeout(() => handleNextQuestionRef.current(), 200);
❌ }, SILENCE_TIMEOUT);

// AFTER:
✅ silenceTimerRef.current = setTimeout(() => {
✅   intentionalStopRef.current = true;
✅   recognitionRef.current?.stop();
✅   advanceTimerRef.current = setTimeout(() => {
✅     advanceTimerRef.current = null;
✅     handleNextQuestionRef.current();

✅   clearTimeout(advanceTimerRef.current);
✅   advanceTimerRef.current = null;
✅ }

[Cell 0]: ✏️  Guard at top of handleNextQuestion() (~line 370)
[Cell 1]: // Add as the FIRST line inside handleNextQuestion():
✅ if (advanceTimerRef.current) {
✅   clearTimeout(advanceTimerRef.current);
✅   advanceTimerRef.current = null;

[Cell 0]: BUG-F   [Low]   File: frontend/src/services/authService.ts  (line ~136)
[Cell 1]: BUG-F   [Low]   File: frontend/src/services/authService.ts  (line ~136)

[Cell 0]: 🔍  Symptom
[Cell 1]: isAuthenticated() returns false for some users, logging them out even though their token has not expired. This causes redirect-to-login loops on page refresh for certain accounts.

[Cell 0]: 🧠  Root Cause
[Cell 1]: JWT tokens use base64url encoding, which replaces + with - and / with _. The standard JavaScript atob() function only handles base64 — it throws a DOMException when it encounters - or _ characters. The try/catch in isAuthenticated() catches the error and returns false, making the user appear unauthenticated. The bug is intermittent because not all tokens happen to contain these characters, which is why it only affects certain users.

[Cell 0]: 🛠  Approach
[Cell 1]: Normalise the base64url payload segment to standard base64 before passing it to atob(). This is a one-line addition.

[Cell 0]: ✏️  Fix in authService.ts (~line 136)
[Cell 1]: // BEFORE:
❌ const payload = JSON.parse(atob(token.split(".")[1]));

// AFTER — normalise base64url to base64 first:
✅ const base64 = token.split(".")[1]
✅   .replace(/-/g, "+")   // base64url → base64
✅   .replace(/_/g, "/");   // base64url → base64
✅ const payload = JSON.parse(atob(base64));

[Cell 0]: ✅  Result
[Cell 1]: isAuthenticated() correctly reads the expiry from all JWT tokens. No user experiences unexpected logouts after page refresh.

[Cell 0]: 🧪  Verify
[Cell 1]: In the browser console after login: run AuthService.isAuthenticated(). It should return true. Then log out, paste in a token with - or _ in its payload (any valid JWT will do), and re-run — it should still return true instead of throwing.
